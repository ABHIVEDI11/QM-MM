# File Reference

A guide to every file type you'll encounter in a QM/MM project, what it
contains, and why it exists.

## Files CHARMM-GUI generates

| File | Contains | Notes |
|---|---|---|
| `step3_input.psf` | Complete topology — every bond, angle, atom type, and charge | This is what `run_qmmm.py` loads as `PSF_FILE`. Never edit by hand. |
| `step3_input.pdb` | Raw coordinates, water on a regular grid | **Do not** use this for QM/MM — water hasn't relaxed yet. |
| `step3_input.crd` | Same coordinates in CHARMM format | Used as input to the equilibration step. |
| `step4_equilibration.pdb` | Coordinates **after** running `openmm_run.py` | This is the file QM/MM should actually use. |
| `toppar.str` | Master file that loads the entire CHARMM36 force field plus your ligand's parameters | Pass to `openmm_run.py` with `-t toppar.str`. |
| `sysinfo.dat` | Periodic box dimensions | Needed for the equilibration run. |
| `<ligand>.rtf` | Ligand topology (atom types, bonds) | Generated by CGenFF/SwissParam from your ligand's structure. |
| `<ligand>.prm` | Ligand force field parameters | Same source as the `.rtf` file; the two always travel together. |

## Files `run_qmmm.py` produces

All written into your configured `output.dir` (default `qmmm_output/`):

| File | Contains |
|---|---|
| `mm_charges.pc` | Every atom that is *not* in the QM region, as a classical point charge `(charge, x, y, z)`. This is what creates the QM region's "feel" of the surrounding protein and water. |
| `qmmm_system.inp` | The actual ORCA input file. The line `%pointcharges "mm_charges.pc"` is the entire QM/MM link — remove it and you get a plain gas-phase DFT calculation instead. |
| `qmmm_system.out` | ORCA's full output. `run_qmmm.py` and `visualize_results.py` both parse this automatically, but you can also `grep` it directly — see below. |

## Useful manual checks

```bash
# Did the calculation finish?
grep "ORCA TERMINATED NORMALLY" qmmm_output/qmmm_system.out

# What was the final energy?
grep "FINAL SINGLE POINT ENERGY" qmmm_output/qmmm_system.out

# What are the Mulliken charges?
grep -A 20 "MULLIKEN ATOMIC CHARGES" qmmm_output/qmmm_system.out
```

A sanity check worth knowing: the most electronegative atom in your QM
region (almost always oxygen, if present) should also carry the most
negative Mulliken charge. If a carbon comes out more negative than every
oxygen in the molecule, something upstream is wrong — check the ligand
charge and the input geometry before trusting any energy number.

## Understanding the energy output

QM/MM gives you one number — the total electronic energy of the QM
region, polarized by the surrounding MM point charges — in three unit
systems:

- **Hartree** — the native unit of quantum chemistry.
- **eV** — common in spectroscopy (1 Hartree = 27.211 eV).
- **kcal/mol** — common in biochemistry (1 Hartree = 627.5 kcal/mol).

**The absolute number is not meaningful by itself.** A QM/MM energy of
`-500 Hartree` tells you almost nothing on its own — it includes the total
electronic energy of every atom in the QM region, not just a binding
energy. What *is* meaningful is a **difference** between two related
calculations, for example:

```
ΔE = E(ligand in protein) − E(ligand in gas phase)
```

That difference isolates the contribution of the protein environment —
electrostatic stabilization, polarization, H-bonding — from the much
larger (and otherwise irrelevant) baseline electronic energy of the
molecule itself.

## Understanding Mulliken charges

Mulliken population analysis assigns a partial charge to each atom in the
QM region, based on how the molecule's quantum wavefunction distributes
electron density. Unlike a classical force field's *fixed* charges, these
respond to environment: the same atom can carry a different charge in
water than in a protein pocket than in vacuum, because the surrounding
point charges genuinely perturb the quantum mechanical solution. That
responsiveness — polarization — is the entire reason to run QM/MM instead
of a faster, purely classical calculation.
